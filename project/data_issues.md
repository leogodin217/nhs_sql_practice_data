# Data Issues

Observations from exploring the NHS dataset. These are features of a simulated dataset, not bugs in your queries. Real data has quirks too -- learning to spot them is part of the job.

1. **FFT recommendation scores go up to 6.** The Friends and Family Test uses a 1-5 Likert scale. A notable proportion of records have score 6 -- a value that shouldn't exist on a 1-5 scale. Run the score distribution query to see how many are affected and what the overall spread looks like.

2. **Many inpatient spells have no discharge.** Count distinct `journey_instance_id` in `fact_admission` vs `fact_discharge` -- you'll see a significant gap. The difference represents "open" spells -- patients admitted but not yet discharged when the simulation ended. This is realistic for a point-in-time snapshot but means ALOS can only be computed on completed spells.

3. **Patient volume ramps up over the dataset's time span.** Early months have fewer admissions than later months. This is partly by design (new patients arrive continuously) and partly a simulation warm-up effect. Keep this in mind for temporal and seasonal comparisons.

4. **`dim_clinic` is an orphan table.** It has 6 rows describing outpatient clinics, but no fact table references it via `entity_id`. You can explore it, but it won't join to anything.

5. **Cancer and outpatient journeys produce multiple events per instance.** A single `journey_instance_id` in `fact_cancer_referral` can have 1 to 100+ rows (due to re-entry loops). Always aggregate to the journey level with `MIN(timestamp)` before computing durations.

6. **Multiple admissions per inpatient spell.** A single `journey_instance_id` in `fact_admission` can have 1-7 rows. Use `MIN(timestamp)` when computing admission time for a spell.

7. **Column names use simulator jargon, not NHS terminology.** The dataset was generated by a generic simulation engine, and three naming conventions leak through into the schema:

   | Column | Simulator meaning | NHS equivalent |
   |--------|-------------------|----------------|
   | `actor_id` | Decision-making agent | **patient_id** |
   | `entity_id`, `entity_type` | Passive resource | **resource** -- ward, consultant, theatre, procedure, etc. |
   | `journey_instance_id`, `journey_type` | Sequence of steps | **spell** (inpatient), **attendance** (A&E), **pathway** (cancer, surgical, outpatient), or **episode** (consultant episode) |

   NHS data teams use specific terms depending on context -- an inpatient stay is a "spell," an A&E visit is an "attendance," and a cancer referral-to-treatment sequence is a "pathway." The schema uses "journey" for all of these. When writing queries, mentally translate: `actor_id` = patient, `entity_id` = the resource involved, `journey_instance_id` = the spell/pathway/attendance being tracked.
